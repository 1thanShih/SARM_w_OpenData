lane_follower

A vision-driven lane following stack for a TurtleBot-class robot. A single USB camera (or two) feeds two computer-vision nodes — one tracking the lane, one watching for triangular turn signs — whose outputs are fused by a Sugeno fuzzy controller into wheel commands sent over rosserial to an Arduino Mega.

Quick start

Three launch files exist for different stages of development.

Single camera (default)

roslaunch lane_follower lane_detect_bringup.launch

Dual camera

roslaunch lane_follower lane_detect_bringup.launch dual_camera:=true

Override camera args inline

roslaunch lane_follower lane_detect_bringup.launch \
    dual_camera:=true \
    lane_cam_device:=/dev/video2 \
    lane_cam_width:=1280 lane_cam_height:=720 lane_cam_fps:=60

lane_detect_bringup.launch

The primary entry point. Brings up the full stack: camera(s), both CV nodes, the fuzzy controller, and the Arduino serial bridge. Every parameter — camera settings, CV pipeline knobs, fuzzy controller gains — is declared inline in the launch file as either an <arg> (camera) or a hard-coded <param> (algorithm). No external config file.

lane_detect_v2.launch

A minimal vision-only launch with no Arduino bridge and no fuzzy controller. Useful for tuning the CV pipeline on a desk — you get the two detectors publishing their output topics, and that's it. All parameters are hard-coded in the launch (no rosparam file).

Topology
hardware / device ROS node topic
No actuation — Nothing subscribes to /lane_detect or /turn_detect in this launch. Use rostopic echo or rqt_image_view to inspect outputs.

A personal tuning launch with the same node graph as the main bringup, also supporting single & dual-camera modes via dual_camera:=true. Carries its own divergent parameter values for trying things without touching the main file.

Nodes in detail

What each node actually does, what it subscribes to, what it publishes, and the parameters that shape its behavior.

camera_node

pkg: camera

Thin wrapper around cv2.VideoCapture. Opens a V4L2 device, applies width/height/FPS via OpenCV capture properties, then publishes each frame as sensor_msgs/Image. Also publishes a base64-JPEG variant on /golfbot/camera_web for legacy web viewers.

script
sensors/camera/src/camera.py
publishes
/camera/image_raw — sensor_msgs/Image (bgr8)
/golfbot/camera_web — std_msgs/String (base64-JPEG)
subscribes
—
params
~camera_id (str, default /dev/video0)
~width (int, default 640)
~height (int, default 480)
~fps (int, default 30)

Behavior notes

  • If the device fails to open, the node still runs but logs a warning and produces no frames.
  • Requested resolution / FPS are passed straight to V4L2; the driver may snap to the nearest supported mode — the node logs the actually applied values.
  • The publish rate is governed by rospy.Rate(fps), so the topic frequency equals the requested FPS even if the camera produces frames faster.

lane_detect_node

pkg: lane_follower

The lane CV pipeline. Takes raw frames and produces a smoothed offset (px from frame center to lane center, signed) and angle (yaw in degrees). The pipeline is six stages: Preprocess → Detect → Track → Measure → Smooth → Render.

script
lane_follower/scripts/lane_detect_v2.py
subscribes
~image_topic — sensor_msgs/Image
(/camera/image_raw single-cam or /cam_lane/image_raw dual-cam; can also be a /dev/video* path to bypass ROS and read straight from the device)
publishes
~pub_topic — lane_follower/LaneData (default lane_detect)
~debug_topic — sensor_msgs/Image (overlay)
~binary_topic — sensor_msgs/Image (mono8 binary mask)
algo params
~strict_roi, ~min_contour_area, ~anchor_ratio, ~lane_width_history_len, ~enable_yaw_b, ~max_predict_frames

Pipeline at a glance

  • Preprocess — gray → Gaussian blur → Otsu threshold (inverted for dark lanes on light floor) → morph open/close → ROI mask.
  • Detect — contour extraction, area filter, plus resolve_floating_v4 which re-merges broken pieces via nearest-distance and fit-line intersection.
  • Track — LaneTracker assigns each contour to left/right by bbox-IoU against the previous frame, falling back to centroid bisection; carries memory for up to 15 frames.
  • Measure — at anchor_y = frame_height × anchor_ratio compute left/right column x in a ±15 px band; yaw via either quadratic polyfit (enable_yaw_b=true) or cv2.fitLine.
  • Smooth — Kalman (default) or EMA over offset & yaw. Predicts for up to 15 frames when measurements are missing, then resets.
  • Render — overlay drawn only when the debug publisher has subscribers (or show_window=true).

turn_detect_node

pkg: lane_follower

Detects triangular turn signs in the frame and emits a stabilized turn direction (left or right) along with the sign's pixel area and horizontal offset from frame center.

script
lane_follower/scripts/turn_detect.py
subscribes
~image_topic — sensor_msgs/Image
(/camera/image_raw single-cam or /cam_turn/image_raw dual-cam)
publishes
turn_detect — lane_follower/TurnDetect
~debug_topic — sensor_msgs/Image (overlay)
params
~debug_topic, ~show_window

Behavior notes

  • Otsu + bitwise_not → approxPolyDP to find 3-vertex contours.
  • Filters: min area 1000 px², aspect ratio in [0.5, 2.0], area/bbox ratio ≥ 0.4.
  • Direction = "right" if the triangle's apex x > bbox center x, else "left".
  • Tracker requires 10 consecutive frames of the same sign before publishing; emits NOT_DETECTED after 5 frames missing.

lane_controller_fuzzy

pkg: lane_follower

Fuses lane offset/angle and turn-sign detections into wheel commands. Uses a Sugeno fuzzy inference (5×5 rule base) for normal lane following and a small priority-based state machine for sign behavior.

script
lane_follower/scripts/lane_controller_fuzzy.py
subscribes
lane_detect — lane_follower/LaneData
turn_detect — lane_follower/TurnDetect
publishes
arduino_vel — geometry_msgs/Twist
params
~base_speed, ~max_angular,
~turn_pixel_threshold_1/2, ~hard_turn_angular_1/2, ~hard_turn_duration_1/2,
~hard_turn_cooldown, ~sign_detect_pixel_threshold, ~sign_offset_threshold, ~sign_align_angular, ~scan_angular_z

Decision priority (high → low)

  • 1. Hard turn in progress — hold a fixed angular velocity until hard_turn_end_time.
  • 2. Sign lost > 0.3 s — stop forward motion and scan left–right (1 s → 2 s → 1 s cycle).
  • 3. Approaching a sign — slow down (base_speed - 0.2) and align if offset exceeds sign_offset_threshold.
  • 4. Default — pure fuzzy lane following, output scaled to ±max_angular.

Fuzzy controller

  • Inputs: offset (5 sets centered at ±100, ±50, 0 px) and angle (5 sets at ±50, ±25, 0°).
  • Output: a 5×5 Sugeno singleton matrix in [−1, 1], defuzzified by Product Inference.
  • Convention: positive output → left turn (so positive offset/angle, meaning car drifted right, yields a leftward correction).

arduino_mega_serial_node

pkg: rosserial_python

Standard rosserial bridge. Forwards geometry_msgs/Twist on arduino_vel down a serial link to the Arduino sketch, which translates linear/angular velocity into motor PWM. Only present in lane_detect_bringup.launch and lane_detect_bringup_Shih.launch.

Configuration

Every value lives in the launch file. Cameras as command-line <arg>s, algorithm knobs as hard-coded <param>s on each node. No YAML, no shared config namespace.

Camera args (overridable on command line)

Declared as <arg> at the top of each bringup launch. Override inline (edit the default) or via roslaunch ... key:=value. See the lane_detect_bringup args table for the full list in the main launch (Shih exposes only dual_camera, lane_cam_device, turn_cam_device).
